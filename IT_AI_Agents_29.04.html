P1.2

Tại sao đây không chỉ là chatbot

Nền tảng có khả năng mở rộng phi tuyến — không build lại từ đầu

REASON 01
Là một nền tảng, không chỉ là chatbot
Team nào cũng tự build & publish agent, không cần qua IT. Shared infra · shared LLM · shared KB · shared security · shared observability. Càng nhiều team dùng, chi phí biên càng thấp.
REASON 02
Một config, đa kênh
Có mặt ngay trên cả myVNG và MS Teams. Nhân viên không cần học thêm công cụ mới — dùng app sẵn có là đủ. Zero friction onboarding cho toàn bộ VNG.
REASON 03
Cho phép từng BU/phòng ban tự quản lý agent
Pilot từng nhóm, đo hiệu quả rồi mở rộng. IT chỉ vận hành nền tảng, không làm thay từng team.
Nền tảng thay vì chatbot đơn lẻ: Không cần build lại từ đầu mỗi khi có nhu cầu mới. Team nào cũng có thể tự tạo agent, tự cập nhật KB, tự publish — IT chỉ vận hành nền tảng. Càng nhiều team dùng, chi phí biên càng thấp.

P2.4

Luồng hoạt động chi tiết — Agent/Provisioning M365

Từ request của nhân viên → account M365 sẵn sàng: dưới 5 phút

Chi tiết 6 bước phê duyệt

  1. Starter chat với IT Agent qua MS Teams / myVNG → LLM thu thập UPN prefix, displayName, mailbox?, end_date.
  2. IT Agent gọi IT MCP Gateway (FastMCP · OAuth · Azure Proxy) → tạo ticket SDP với template "Service Account Request".
  3. SDP webhook → IT Agent gửi Adaptive Card + LLM Insights đến HoD trong MS Teams.
  4. HoD approve/reject trên Card hoặc reply tự nhiên → LLM parse intent → IT Agent cập nhật SDP.
  5. SDP "Final Approved" → webhook đến Agent/Provisioning M365 → 16 phases → account ready trong <5 phút.
  6. Agent dùng IT MCP GW ghi audit trail ngược về SDP — SDP là nguồn duy nhất chứa toàn bộ vết tích.
P2.5

Mô hình bảo mật — Defense in Depth (7 Layers)

Mỗi lớp độc lập, tự đủ chặn 1 attack vector · Layer 5+6 là cốt lõi bảo vệ

7 lớp bảo mật độc lập hoàn toàn. Kể cả attacker nắm toàn bộ certificate, chỉ thực hiện được tối đa 6 operations cứng trong code. Layer 5 + Layer 6 là bộ đôi cốt lõi — compromise một lớp không kéo theo lớp kia.
⭐ Layer 5 — Code Whitelist: Chỉ 6 Graph operations được phép theo list tĩnh. Mọi call khác (DELETE, PATCH users…) đều bị block tại code level — không phụ thuộc runtime permission.
⭐ Layer 6 — Restricted AU: Sau khi user được add vào AU, ngay chính agent tạo ra nó cũng mất quyền PATCH/DELETE user đó vĩnh viễn. Bảo vệ tại Microsoft Entra platform.

Lộ trình triển khai

4 phases có exit criteria rõ ràng · Rollback < 30 giây bất kỳ lúc nào

⏳ Pending để go-live Phase 3:
(1) IT Agent pentest security hoàn thành & sign-off
(2) MCP Gateway (FastMCP · OAuth · Azure Proxy) production-ready
(3) IT Security threat model review + sign-off · Conditional Access policy cho Service Principal
🔴 Kill switch: Set AGENT_DISABLED=true → redeploy → toàn bộ capability bị tắt trong < 30 giây. Rollback bất kỳ phase không cần thay đổi code hay database.
